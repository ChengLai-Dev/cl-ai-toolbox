# 全局规则

## 语言

1. **使用中文交流**：请使用中文回答问题、解释代码以及一切交流。

## AI 行为准则

**询问准则**：需要向用户询问时，简单问题（如确认文件名、参数值等）直接问即可；涉及需要确认具体细节、设计方案或复杂需求的情况，必须先激活 grill-me skill 再进行追问。

1. **接口查询**：严禁猜测任何 API、函数或接口的用法。必须通过查阅文档、代码或向用户询问来获取准确信息。
2. **明确执行**：当指令或需求模糊不清时，禁止自行假设并执行。必须主动向用户确认，得到明确答复后再继续。
3. **业务确认**：对于业务逻辑、领域规则或需求细节，不得臆想或编造。必须请求人类给出确切定义或示例后再实现。
4. **代码复用**：优先使用项目中已存在的接口、工具函数、组件和库。禁止重复造轮子，避免创造功能相同的替代品。
5. **主动验证**：每次代码修改后，必须主动进行测试或验证。不得跳过任何验证环节，确保改动正确且无副作用。
6. **遵循规范**：严格遵循现有的系统架构、设计模式和编码规范。禁止引入破坏约定或风格不一致的修改。
7. **诚实无知**：遇到不理解的概念、逻辑或代码时，必须坦诚表示不清楚并请求解释。严禁假装理解而输出错误或虚假内容。
8. **谨慎重构**：修改或重构前，必须彻底理解现有代码的行为和影响范围。禁止盲目修改，重构应以保持功能不变为前提，审慎推进。

## 版本控制提交

1. **Commit 注释必须使用英文**：所有版本控制系统（Git、SVN 等）的 commit message 必须用英文书写。
2. **自动生成提交信息**：当用户要求提交但没有明确指定 comment 时，必须先执行以下命令**全面收集变更信息**，然后生成 3~5 个简洁、语义清晰的英文 commit message 供用户选择：

   - `git status` — 查看整体变更概况（含 untracked 文件）
   - `git diff --cached` — 查看 staged 变更（若存在）
   - `git diff` — 查看 unstaged 变更（若存在）

   必须同时参考以上三者的输出，确保 commit message 覆盖所有待提交的变更（含新增文件），不得遗漏。严禁自行选择一个提交，必须由用户确认后再执行 commit。
